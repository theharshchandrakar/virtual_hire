# 07 — Technical Stack

**Purpose:** Lock in concrete technology choices for each architectural component, with rationale and the rejected alternative.

**Depends on:** [06-architecture.md](06-architecture.md) (components this stack implements).
**Feeds into:** [08-privacy-and-compliance.md](08-privacy-and-compliance.md) (storage/hosting choices affect compliance posture) and [09-roadmap.md](09-roadmap.md) (phasing assumes this stack's capabilities).

> **Revision note (2026-07-15):** the vector database choice changed from pgvector-inside-Postgres to a dedicated Qdrant instance. This is a direct reversal of the "no dedicated vector database" decision this document previously made — see below for the honest accounting, and [CHANGELOG.md](../CHANGELOG.md) for when this changed.

> **Revision note (2026-07-16):** all LLM crew model access moves to **OpenRouter** as a unified gateway, replacing direct Anthropic API calls, and a new large (200–300B-parameter class) Verdict/Judge model joins the crew for the three scored-assessment services in [00-ideation.md](00-ideation.md). New integration points (video platform, proctoring signal detection) are added with several choices explicitly left open — this revision states what's decided and flags what isn't, rather than guessing at vendor names.

## Stack choices

| Layer | Choice | Why | Alternative considered |
|---|---|---|---|
| Backend language/runtime | Python 3.12 | RAG and multi-agent LLM orchestration are core, not a delegated side call — Python has the deepest ecosystem for both. | TypeScript/Node (original choice) — rejected once the system's central logic became RAG retrieval and multi-agent orchestration. |
| Backend framework | FastAPI | Async-native, Pydantic models map directly onto structured LLM outputs and RAG payloads, automatic OpenAPI schema for the frontend. | Django — heavier framework overhead not justified for an API-first, orchestration-heavy backend. |
| Frontend framework | Next.js (React) — unchanged | Pure API client of the FastAPI backend; right fit for a data-dense HR dashboard. | A Python server-rendered framework — rejected, would recouple frontend to backend language. |
| Primary relational database | PostgreSQL | ACID-transactional storage for the pipeline/scorecard/audit data whose invariants (I3–I9) depend on relational constraints, triggers, and row-level security. No longer holds any vector data — see the Vector database row below. | MySQL — rejected on the same grounds as the original stack decision (RLS + JSONB + enum-type support favor Postgres). |
| Vector database | **Qdrant** (one collection per Organization) | Purpose-built ANN performance and richer filtering/hybrid-search headroom than pgvector offers, without that workload competing with the primary OLTP Postgres instance as resume volume grows; lets the vector workload (high write volume from chunking, different query shape) scale independently of the relational workload. Hosted as **Qdrant Cloud** (managed) rather than self-hosted, to stay consistent with this stack's existing "avoid owning infrastructure that's undifferentiated relative to the product problem" posture (same reasoning already applied to RDS and ElastiCache below). | **pgvector inside the existing PostgreSQL instance** — this was the v1 decision as of the 2026-07-14 pivot, and is the alternative being explicitly rejected in *this* revision. That decision's stated rationale (avoiding a second tenant-isolation surface) was sound and is not dismissed lightly — see the "collection-per-organization" isolation design in [06-architecture.md](06-architecture.md) for how the reintroduced surface is mitigated. Self-hosted Qdrant (e.g., on Fargate + EBS) was also considered and rejected for the same "avoid operating a new stateful system ourselves" reasoning this stack already applies elsewhere. |
| Embeddings model | Voyage AI (`voyage-3`, 1024-dim) — unchanged | Purpose-built retrieval embeddings with strong domain recall. | Self-hosted open-source embedding models — rejected for v1, avoids standing up embedding inference infrastructure. Not moved to OpenRouter — OpenRouter is an LLM (chat/completion) gateway, not an embeddings provider; Voyage stays a direct integration. |
| LLM crew / agent orchestration | CrewAI — unchanged | Purpose-built for role-based agents collaborating as a defined crew, each bindable to a different model by task complexity. | LangGraph — rejected for v1, requires materially more custom orchestration code. |
| **LLM provider gateway** | **OpenRouter**, accessed via LiteLLM's built-in `openrouter/` provider prefix — LiteLLM is already a transitive dependency of CrewAI (see `requirements.txt`), so this is a model-string and credential change (`OPENROUTER_API_KEY` replacing `ANTHROPIC_API_KEY`), not a new SDK dependency | One provider integration and one API key for every crew model, including models from providers Sift doesn't have a direct relationship with (needed for the new Verdict model below, which may not be an Anthropic model at all); model swaps for any crew role become a config change, not a code change | **Direct Anthropic API access** — this was the v1 decision before this revision, and is the alternative being explicitly rejected here. Direct access has lower per-call latency overhead and one fewer hop of operational dependency, which is a real cost of this switch, not a free upgrade — see Open Questions. |
| LLM models (multi-model assignment) | Claude Haiku 4.5 — Extraction Agent; Claude Sonnet 5 — Summarizer Agent; Claude Opus 4.8 — Reasoning Agent; **all three now addressed via their OpenRouter model IDs (`openrouter/anthropic/claude-...`) rather than called directly** | Matches model capability and cost to task complexity and call volume — unchanged reasoning, changed transport. | A single model for every crew role — rejected: wastes spend on high-volume extraction or under-powers high-stakes reasoning. |
| **Verdict/Judge model [New]** | A large open-weights model in the **200–300B-parameter class**, accessed via OpenRouter — **exact model not yet named; this is an open decision, not a placeholder for something already chosen** | The three scored-assessment services in [00-ideation.md](00-ideation.md) need a model capable of reviewing structured, multi-source input (Scoring Engine output + resume/transcript/proctoring-event context) and producing a calibrated, well-reasoned narrative verdict — a class of model distinct from, and likely larger than, the existing per-task crew models. Routing it through OpenRouter rather than a direct provider integration is specifically what makes "pick any 200-300B-class model available on OpenRouter" a config decision rather than a new vendor integration. | A frontier proprietary model (e.g., another Claude tier) for this role — not rejected outright, just not what "another 200-300B parameter LLM" in the product brief specifies; revisit once a specific model is named. |
| Async task queue | Celery + Redis — unchanged | Mature multi-step task chaining for the parse → embed → (on-demand) crew pipeline, now also chaining into proctoring/transcript ingestion and verdict generation. | RQ — lacks Celery's chaining/canvas features; Dramatiq — rejected mainly for ecosystem/hiring-familiarity reasons. |
| Object storage | S3-compatible storage (AWS S3) — unchanged, scope extended to assignment submission files | Standard for file storage with signed-URL access patterns. | Storing files as DB blobs — rejected outright. |
| Authentication (HR users) | Managed auth provider (Auth0/Clerk) with org-scoped sessions — unchanged | JWT validation from FastAPI is no harder than from any other framework; org_id in the session drives both the Postgres RLS variable and Qdrant collection resolution. | Custom-built auth — rejected for security-surface reasons. |
| Authentication (candidates) | Magic-link / time-limited email token, no password — unchanged | Matches A9 in [02-assumptions.md](02-assumptions.md). | Full candidate account system — still out per the Scope Creep Watchlist. |
| **Video platform integration [New]** | Ingest via the platform's own webhook/bot/recording-pull API (Zoom, Google Meet, or Teams) — **which specific platform(s) v1 targets first is an open decision, not yet made**; a platform-agnostic third-party meeting-bot vendor is also a candidate, trading a single integration for a dependency on that vendor's own platform coverage | Matches A21/A28 and I15 — Sift never hosts or joins the call as an active participant, only observes/ingests. | Building direct, deep integrations with all three platforms simultaneously — rejected for v1 scope; pick one first, per the open question in [01-problem-space-and-scope.md](01-problem-space-and-scope.md). |
| **Proctoring signal detection [New]** | **Buy, not build** — integrate a third-party face/gaze/voice-detection vendor's API rather than developing computer-vision models in-house; **exact vendor not yet named, an open decision** | Matches this stack's existing "no self-hosted ML/NLP infrastructure" posture (see below) — building biometric detection models from scratch is a far larger, more specialized undertaking than anything else in this stack, with its own accuracy/bias-audit burden. | Self-built detection models — rejected for the same reasoning that already rejected self-hosted embedding/LLM inference elsewhere in this stack, applied to a domain (biometric CV) with materially higher build cost and liability than text embeddings. |
| **Speech-to-text (transcript fallback) [New]** | Prefer the video platform's own transcript feature (no new vendor) where available; **fallback STT vendor for platforms/meetings without one is an open decision** | Matches A27 — avoids standing up STT infrastructure when the platform already provides it. | Always generating transcripts via a dedicated STT vendor regardless of platform support — rejected as unnecessary spend/complexity when the platform-provided path covers most cases. |
| Hosting | AWS: ECS Fargate for the FastAPI API and all Celery workers (separate task definitions per worker type, now including Proctoring Analysis and Transcript & Assignment Reviewer workers), RDS for PostgreSQL, S3 for objects, ElastiCache for Redis, **Qdrant Cloud for the vector store** | Same managed/serverless-adjacent posture across every stateful dependency. | Self-managed Kubernetes, and self-hosted Qdrant specifically — both rejected for the same operational-overhead reasoning applied elsewhere in this stack. |
| Email delivery | Transactional email API (Postmark/SES) — unchanged | Deliverability/bounce handling is a solved problem. | Self-hosted SMTP — still rejected. |

## Choices made specifically to keep v1 scope tight

- **No microservices split, despite the added storage surface.** The API and all Celery workers remain one Python codebase, one repo, one data-access layer, even though they now talk to two stateful storage systems (Postgres, Qdrant) instead of one.
- **A dedicated vector database (Qdrant), scoped by collection-per-organization rather than a shared collection with a payload filter.** This is a deliberate reversal from the prior "no dedicated vector database" position — restated honestly here rather than silently dropped: the isolation-surface cost the original decision was trying to avoid is real, and is mitigated (not eliminated) by giving Qdrant a structural per-organization boundary instead of relying on filter discipline alone.
- **No self-hosted ML/NLP or embedding infrastructure.** Both the LLM crew and the embedding step call hosted provider APIs (Anthropic, Voyage) rather than standing up model-serving or GPU infrastructure. Qdrant Cloud extends this same posture to the vector store itself — it is a managed database service, not inference infrastructure, so hosting it managed rather than self-managed is a direct application of this principle, not an exception to it.
- **No custom auth.** Unchanged rationale from the original stack decision.
- **A fixed per-task model assignment via one gateway, not a configurable/pluggable model registry.** Extended from the prior "fixed three-model crew assignment" decision — OpenRouter makes *which* model fills each fixed role a config change, but v1 still ships with one model per role, not a registry an HR user or org admin can choose from.
- **Buy, not build, for biometric/CV detection and (where possible) transcription.** New in this revision, extending the existing "no self-hosted ML/NLP infrastructure" principle to a domain (computer vision, speech) this stack hadn't previously needed.

## Open Questions

- Does the LLM provider layer need a fallback/secondary path for availability, especially given a live user-facing search flow depends on it synchronously-from-the-user's-perspective? **Compounded by this revision:** OpenRouter itself is now also a single point of failure sitting in front of whichever underlying provider(s) serve each crew role — a fallback question that existed before this revision (no secondary LLM provider) now has an extra hop to reason about, not fewer.
- Should CrewAI's agent definitions live in the same FastAPI/Celery codebase, or is there a case for packaging the crew as an independently versioned/deployed component?
- At what per-organization chunk volume, or total collection count, does Qdrant Cloud's plan/tier need reassessing — does collection-per-organization remain cost-effective at higher organization counts than [02-assumptions.md](02-assumptions.md)'s A14 anticipates, or does the shared-collection-with-payload-filter multitenancy pattern Qdrant itself recommends become necessary?
- Is Voyage AI's embedding dimension (1024) and model version something to pin with a documented migration plan — changing it later requires provisioning new Qdrant collections (vector size is fixed per collection) and re-embedding every point across every organization.
- **New in this revision:** does Qdrant Cloud's authentication model (API key scoping) support per-collection or per-organization scoped keys, which would add a fourth isolation layer beyond collection-per-org + payload filter + application-layer session scoping — worth evaluating once collection-per-org volume is known, not a v1 blocker.
- **New 2026-07-16:** what is the exact Verdict/Judge model? "200-300B-parameter class, via OpenRouter" is a class of model, not a pinned choice — this needs a specific OpenRouter model ID before E17 can be implemented, and should be re-evaluated periodically since OpenRouter's catalog changes over time.
- **New 2026-07-16:** which video platform(s) does v1 actually integrate with — see the identical open question in [01-problem-space-and-scope.md](01-problem-space-and-scope.md); this stack doc can't lock down a specific API/SDK dependency until that's answered.
- **New 2026-07-16:** which proctoring signal detection vendor — evaluation criteria should include not just accuracy but the vendor's own bias-audit/fairness documentation, given the disparate-impact risk this stack has otherwise been careful about (see the autonomous-ranking exclusion in [01-problem-space-and-scope.md](01-problem-space-and-scope.md)); this is as much a compliance decision as a technical one, and should involve [08-privacy-and-compliance.md](08-privacy-and-compliance.md)'s review process, not be made as a pure build-vs-buy engineering call.
- **New 2026-07-16:** does OpenRouter's own data-handling/retention policy for request content need the same subprocessor DPA review as Anthropic/Voyage/Qdrant Cloud already require — see [08-privacy-and-compliance.md](08-privacy-and-compliance.md).
